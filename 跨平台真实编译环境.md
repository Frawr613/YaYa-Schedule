# 鸦鸦日程跨平台真实编译环境

## 当前建议

鸦鸦日程现在最适合采用“共享 Web 核心 + 双端原生壳”的方案继续开发：

- Windows 本地：负责 Web 预览、共享资源编辑、Android APK 编译。
- GitHub Actions macOS：负责 iOS/Xcode 真编译。
- 真机 iOS 发布：需要 Apple Developer Team、证书和描述文件后再接入签名流水线。

这样不需要推翻现有十层架构，也能让 iOS 每次提交都经过真实 Xcode 编译。

## 已部署的真编译能力

仓库新增：

```text
.github/workflows/ios-compile.yml
```

它会在以下情况运行：

- 推送到 `main`
- Pull Request
- 手动点击 GitHub Actions 里的 `Run workflow`

流水线会执行：

1. 检查 Xcode 版本和模拟器运行时。
2. 检查 Web 资源语法和缓存版本标记。
3. 检查 plist / entitlements。
4. 使用 `xcodebuild` 编译 `YayaScheduleClean` 主 App。
5. 使用 `xcodebuild` 编译 `YayaScheduleWidget` Widget 扩展。
6. 上传 iOS Simulator 编译产物和日志。

这是实际 Xcode 编译，不是本地静态检查。

## 关于 iOS 签名

当前流水线使用：

```text
CODE_SIGNING_ALLOWED=NO
```

这能验证 Swift、Widget、Web 资源和 Xcode 工程是否能真实编译到 iOS Simulator。

如果要生成可安装到真机或 TestFlight 的 `.ipa`，还需要：

- Apple Developer Team ID
- Distribution Certificate
- Provisioning Profile
- App Groups 对应的签名能力

拿到这些材料后，可以继续增加签名 Archive workflow。

## 可选的长期跨平台方案

如果以后想进一步统一双端原生桥，可以考虑 Capacitor：

- 保留现有 Web 十层结构。
- 用 Capacitor 承载 Android/iOS WebView。
- 将通知、教务导入、Widget、App 图标等写成平台插件。

但这属于架构迁移，不适合在当前功能修复密集阶段立刻替换。当前优先级应当是先让现有双端都能真实编译、可验证、可回滚。

## GitHub 保留范围

当前 GitHub 仓库只保留 iOS 端真实编译需要的内容：

- `YayaScheduleClean.xcodeproj` Xcode 工程。
- `YayaScheduleClean/` 主 App 源码、Web 十层结构资源、图标和配置。
- `YayaScheduleWidget/` iOS Widget 扩展源码和配置。
- `.github/workflows/ios-compile.yml` iOS 真实编译流水线。
- `README.md`、`iOS实机检验清单.md` 和本说明文档。
- `dist/鸦鸦日程-iOS-Xcode工程包-2.0.zip` 维护中的 iOS Xcode 工程包。

以下内容只允许留在本地，不进入 GitHub：Android APK/AAB、Android 工程、Widget 预览截图、历史时间戳压缩包、Xcode DerivedData 和本地构建缓存。
